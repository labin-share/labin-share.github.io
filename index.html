RedLock的不足之处

redlock的不足之处讨论 Martin Kleppmann

RedLock 不能完全保证正确性，其根本原因出现在RedLock的锁有过期时间的概念。

由于系统本身有阻塞，调度，系统时间阶跃的问题，会对过期时间产生一些预期之外的事情：

  1. 获得锁之后，客户端在执行任务的期间，有可能会发生阻塞或者系统本身的调度
    • 如果在阻塞挂起期间，锁已经过时，然后任务恢复运行并没有感知锁已经过时，继续执行任务，将会发生错误。
  2. 系统时间阶跃的问题
    • 系统阶跃问题指的是，系统的时间可能发生不连续的跳跃，可能会向前也可能会向后。而redis的过期时间是基于系统时间的，如果向前跳跃，可能锁的过期时间会因此而较早结束。
    • 基于上面的原因，假设有A,B,C三个redis server，本来客户端1已经从A，B获得了锁，但是因为时间阶跃的问题，导致B早些过时，导致B，C是无锁的状态，那么另外一个进程就可以从这两个redis server获得锁，导致系统同时存在客户端1和客户端2同时拥有锁，在执行任务的状态。

分布式锁-zookeeper实现

实现原理
curator官网-能实现的功能

1. zookeeper分布式锁实现原理

zookeeper有一种临时类型的节点和顺序节点，并且节点删除的时候会通知客户端。可以利用这些特性，做成分布式锁。

  1. 进程1向zookeeper创建一个节点，该节点的编号假设是5
  2. 拿出所有的节点，判断当前节点是否是最小节点
    1. 如果是，则获取锁
    2. 如果不是，则让该节点注册watcher，检测前一个节点有没有被删除。并且进行休眠，知道节点删除，在watcher的回调里面，唤醒休眠的线程。

1.1 为什么是临时类型的节点

临时节点有一个特点， 当客户端因为某些原因跟zk断开的之后，由于zk没有收到客户端的心跳，会认为客户端失效，从而会删除临时节点。这样能够避免因为线程或进程挂掉导致锁无法释放，造成死锁的情况。

1.2 为什么是顺序节点

当进程创建了一个节点之后，发现不是最小的节点，那么需要监控前一个节点，如果节点是有序的，那么可以直接的找到被监控的节点。

2. zookeeper实现分布式锁例子

从maven repository下载zookeeper依赖

该例子模拟curator实现了一个分布式锁类

org.apache.curator curator-recipes 2.6.0

package com.study.test;

import org.apache.curator.framework.CuratorFramework;
import org.apache.curator.framework.CuratorFrameworkFactory;
import org.apache.curator.framework.recipes.locks.InterProcessMutex;
import org.apache.curator.retry.ExponentialBackoffRetry;
import org.junit.Test;

public class CuratorTest {
@Test
public void process1() throws Exception {

    CuratorFramework cf = CuratorFrameworkFactory.builder()
            .connectString("192.168.99.100:2181")
            .retryPolicy(new ExponentialBackoffRetry(1000, 3))
            .sessionTimeoutMs(2000)
            .build();
    cf.start();

    InterProcessMutex lock = new InterProcessMutex(cf, "/distribute-lock");
    lock.acquire();
    System.out.println("process1 got the lock, do something");
    Thread.sleep(10000);
    lock.release();
}

@Test
public void process2() throws Exception {

    CuratorFramework cf = CuratorFrameworkFactory.builder()
            .connectString("192.168.99.100:2181")
            .retryPolicy(new ExponentialBackoffRetry(1000, 3))
            .sessionTimeoutMs(2000)
            .build();
    cf.start();

    InterProcessMutex lock = new InterProcessMutex(cf, "/distribute-lock");
    lock.acquire();
    System.out.println("process2 got the lock, do something");
    lock.release();
}

}

该例子模仿了利用curator分布式锁同步两个进程process1, process2的例子。

process1获取到锁之后，休眠10秒钟，用于模拟任务。这时候开启process2，因为process1的锁还未释放，所以process2处于等待中，知道process1锁释放。

简单来说，redis的锁的实现比较简单，性能页比较高效，但是zookeeper来实现分布式锁要比redis安全可靠。
