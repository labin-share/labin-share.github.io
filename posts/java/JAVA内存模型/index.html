JAVA内存模型
2019-08-01

1. 多处理器系统

现代操作系统的内存模型

  • 每个处理器都有自己的缓存, 每个处理器的缓存会根据一些协议更新到主内存。
    • 这样会面临一个问题，缓存一致性。即每个处理器的缓存怎么同步都主内存，保存哪个处理器的缓存版本，让其他的处理器看到。
  • 处理器会对代码乱序执行 （跟jvm编译器重排序的概念类似，不过是不同的东西）
    • 乱序执行指的是处理器为了提高执行的效率，可能会打乱执行代码的顺序，跟输入的代码的顺序不一致。
    • 但是这会导致在多线程的情况下，如果一个线程依赖另一个线程的执行，会由于代码顺序被打乱，导致结果不正确。==举例子==

缓存一致性的协议有很多，MSI， MESI等等。不同的操作系统会有一些区别。

2. JAVA 内存模型 (Java Memory Model,JMM)

由于不同操作系统和硬件的内存模型是不一样的。这会导致java程序在多线程的情况下，在不同的操作系统上运行的情况不一致，在一个平台上运行正常，可能到另一个平台就出错了。
为了屏蔽不同操作系统和硬件的内存模型的区别，JVM定义了自己的内存模型。

JAVA内存模型是一组规范，定义了程序中的一些线程共享的变量的访问。

这组规范主要的内容描述是：

  • 主内存
    • 所有的共享变量都存储在了主内存中。线程不能直接访问主内存的变量。主内存对应到jvm的堆。
  • 线程的工作内存
    • 线程工作的内存区域，线程之间不能互相访问对方的工作内存。工作内存对应着线程栈。
    • 工作内存存储的变量是共享内存中的副本。在对变量进行操作的时候，线程会拷贝一个副本到工作内存中，如果不进行特殊的控制 (比如synchronize，volatile等)，对副本的修改不会立即同步到主内存。可能会导致每个线程工作内存的副本有所区别。
    • 线程要通过更新副本到主内存来保持变量的同步。
    • 一般情况下，JMM为了效率，会直接让线程读取工作内存的值，而不会重新同步主内存到工作内存。
  • 线程的工作内存和主内存交互的操作
    • JMM为了实现工作内存和主内存的交互，定义了一组操作。
    • lock, unlock, load, read, use, assign, store, write。这些操作本身都是原子性的。
    • 这些操作之间相互有一些约束，比如对一个变量的load和read必须成对出现 （但不保证是连续出现）；对一个变量lock的时候会重新读取 *（load, read）主内存的变量到工作内存，unlock的时候会写入 *（store, write） 到主内存等等。

2.1 例子-演示各个线程副本值不一致的情况

如下例子，创建了两个线程。

  1. 线程1先执行，并且等待两秒之后再执行线程2
  2. 线程1中需要等待stop=true的时候才会停止。
  3. 线程2中执行后会设置stop=true让线程1停止。
  4. stop是两个线程共享的变量。

但是从运行的结果可以看到， 线程2停止了之后，线程1还是没有停止，但是这时候线程2已经设置了stop = true。只是线程1工作内存中stop=false，所以线程1没有按预想的停止。

2.2 JMM数据原子操作

JMM定义了一系列的原子操作用于操作数据。比如从主内存中读取数据到线程的工作内存中。

  1. read。 从主内存中读取数据，但是还未加载到线程的工作内存中。
  2. load。将读取出来的数据加载到工作内存中。
  3. use。cup从工作内存中拉取数据进行计算。
  4. assign。将计算之后的数据赋值到工作内存中。
  5. store。将工作内存的数据写入主内存，但是还未修改原来的变量值。
  6. write。将store到主内存的数据赋值给主内存的变量。
  7. lock。
  8. unlock

2.3 MESI缓存一致性协议

当某个cpu修改了高速缓存中的数据时，该数据会马上同步回主内存，其它cpu通过总线嗅探机制可以感知到数据的变化从而将自己高速缓存中的数据失效，再次从主内存中拿取。

每一个cpu是通过跟cpu总线跟主内存进行交互的。

cpu总线嗅探机制：类似于监听机制，不断的监听变量值的变动。当有共享变量被修改了，线程将会让自己工作内存中的变量失效，要使用的时候需要从主内存重新拿取

缓存一致性协议解决了，每个线程自己修改了工作内存之后，虽然同步回了主内存，但是其它线程无法感知到主内存值变化的情况。

2.4 JMM是如何才能保证可见性的

如果需要保证某个线程对变量的赋值立马对其它线程可见，需要同时满足以下两个条件：

  1. 采取某种机制，让cpu缓存中的变量修改的值能立即同步到主内存， 比如利用volatile关键词。
  2. 利用MESI缓存一致性协议，使得其它线程能够感知到值的改变

这两个机制共同保证了可见性。虽然MESI可保证变量的值改变之后能够同步到主内存，让其他的线程感知到。但是指令在执行的时候会有重排序的问题，并不能保证变量在同步到主内存之前不会被其他操作穿插。所以JMM这之上做了一些处理，比如volatile。

比如，在程序中，先对A赋值，再对B赋值，虽然A最终会同步到内存并且被其他线程感知到，但是很有可能B先于A同步到主内存并且被其他线程感知。

JMM的什么时候同步主内存和工作内存：如果不加任何的控制，线程对变量进行改写时 (assign)， 之后会对主内存进行同步 (store, write)。但是改写之后什么时候会写道主内存时不一定的，但是在线程结束之前一定会同步。

2.5 JMM是如何屏蔽区别的

在JMM定义的这套规则之下，再加上java提供的一些语义，比如volatile， synchronized， Lock等等，就可以让用户自己决定怎么保证线程的原子性，可见性，有序性，而无需考虑java程序运行的平台是什么，只需要考虑JMM即可。

比如可以使用synchronized块来保证一组操作的原子性，再操作未完成前不会有其他线程的打扰，JMM令其在所有平台下运行的结果都是一致的。

2.6 先行发生原则

先行发生原则，指的是JMM定义的，两项操作之间的顺序关系。简单的来说，就是如果操作A先于B被cpu执行，那么操作A的影响应该能被B观察到。

先行发生原则只是一些基础的原则，可以减少代码的繁琐，但是不能保证所有的情况都是线程安全，所以才需要锁等技术进行补充。

先行发生原则有以下8种：

  • 程序次序规则
  • 管程锁定规则
  • volatile变量规则
  • 线程启动规则
  • 线程终止规则
  • 线程中断规则
  • 对象终结规则
  • 传递性规则

3. volatile

volitile是java最轻量级的同步机制。

它保证了数据的：

  1. 可见性。使得数据的修改立即同步到主内存。然后再加上缓存一致性协议的作用，使得赋值操作立马对其它线程可见。
  2. 有序性。禁止了重排序 （编译器优化重排序等） 。

按照我的理解，volatile主要是解决了有序性的问题，由于解决了有序性的问题，才保证了可见性。

3.1 volatile 保证有序性的底层原理

3.1.1 重排序

重排序应该分为三种：

  • 编译器优化重排序：编译器在不改变单线程程序语义的前提下，可以重新安排语句的执行顺序。
  • 指令级并行的重排序：如果不存在数据依赖性，处理器可以改变语句对应机器指令的执行顺序。
  • 内存系统的重排序：处理器会使用写缓冲区，会将更改的变量先放在缓冲区中，之后在保存的时候有可能会乱序。

3.1.2 保证有序性的底层原理

经过重排序之后的代码，在执行的时候可能会导致出错。而volatile指令可以禁止这种重排序。

volatile通过内存屏障的作用：

  1. 禁止了编译器优化重排序和指令级并行的重排序。从而保证了指令按照程序的顺序执行。
  2. 禁止了内存系统的重排序。从而保证了先更改的值会立即同步到内存中，而不会被其他的值抢先。
    • volatile变量的load, read, use以及assign, store, write必须是连续执行的。所以才保证了更改的值不会被其他的操作打断。
    • 比如对A进行赋值的时候，assign,store,write是连续执行的，所以会立即同步到内存。

3.1.3 volatile如何提示需要禁止重排序

内存屏障分为四种：

  1. LoadLoad。 表示volatile变量的读取操作会先于其后的读取操作起作用。
  2. StorSstore。 表示volatile变量的写入操作会先于其后的写入操作起作用。
  3. LoadStore。 表示volatile变量的读操作会先于其后的写入操作先起作用。
  4. StoreLoad。 表示volatile变量的写入操作会先于后面的读取操作先起作用。

volatile就是在编译的时候在指令前后加入了这几种屏障指令，编译器和处理器会根据屏障指令来进行相关操作。

3.1.3 保证可见性的原理

正是因为保证了有序性，再结合MESI，保证了更改的变量会对其他线程立即可见。

3.1 volatile保证可见性的底层原理

volatile对JMM的几个操作加了几个约束：

  • load, read, use必须连续执行
  • assign, store, write必须连续执行。即。

volatile规定assign, store, write必须连续执行，这就保证了，一旦线程A对volatile变量进行赋值操作， 那么该操作修改缓存，同步到主内存，修改主内存的值是连续执行，不会被其他的线程执行的操作打断。那么其他线程在这之后进行读取的话，将会是最新值。

3.3 volatile不保证原子性

volatile 对于原子性的赋值操作能够保证其可见性，保证多线程在对同一个变量进行赋值的时候不会出错。但是对于非原子性的操作还是会出错的。

非原子性的操作是有多个操作的，这些操作是分开进行的，而真正影响变量值的只有最后的赋值操作。当操作分开的时候，就有可能导致，线程1和线程2都做好了计算，如果线程1先于线程2做了赋值操作，虽然线程1的值同步到了主内存，也不会导致线程2获取最新值，因为线程2已经计算好了结果。最后线程2进行赋值操作，就覆盖了线程1的值。

即如果从底层原理来看的话，假如一个 i++ 的操作，加1的操作是load,read,use的组合，volatile能够保证它是连续执行的。赋值是assign, store, write,volatile也能保证它是连续执行的。但是volatile不能保证i++这两个操作是连续执行的，即不能保证load,read,use，assign, store, write是连续执行的。所以就算线程1进行了赋值操作，只能保证下一个线程在load的时候从主内存拿最新值，但是不能保证在次之前的线程会拿最新值。所以才会出现错误。如图所示

  1. 假设线程1和线程2同时做了load,read,use的计算
  2. 然后线程1先于线程2进行赋值，此时主内存更新为2
  3. 然后线程2由再次赋值，变量值还是2

3.3.1 例子

开启10个线程，每个线程都对同一个volatile变量进行加一操作，循环1万次，最终结果理想是10万。但是实际上结果会小于十万。原因是

  1. count++是两个操作， 加一计算和赋值。真正修改变量的是赋值操作。
  2. 所以容易出现，线程1和线程2都是读取和变量值0去做计算。
  3. 但是在赋值的时候， 线程1先给变量赋值，变成了1，同时同步到了主内存。
  4. 后面读取的线程都会拿到最新值，但是线程2在变量改变之前已经计算好了，并不会受到影响。所以线程2对变量进行修改之后，变量的值还是1。这就导致了，原本经过两个线程的计算，应该值为2的，但是现在只有1。
