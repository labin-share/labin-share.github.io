1. Sharding JDBC

Sharding JDBC是一个用于在分库分表的环境下，对数据进行操作的框架。
它的作用是，通过一些配置，就可以在代码中操作多个分库，多个分表，这个过程是透明化的。比如执行一条in查询语句的的时候，会去到多个不同的分库或者分表中去查询汇总数据，但是不会影响sql语句的写法，依旧是跟原来的一样。

比如有一个简单的表，里面有两个字段，user_id, user_name。假设是按照user_id字段去分片，偶数的分到数据库1，奇数的分到数据库2。

2. Sharding JDBC的相关概念

Sharding JDBC相关概念

2.1 表相关

2.1 逻辑表

一张表的总称，在数据库中并不存在，只是从逻辑上来讲。比如对于my_table这张表，在数据库中有两张分表，my_table0和my_table1，那么这两张表的逻辑表就是my_table。

在程序中编写代码的时候，只需要面对my_table进行编程即可，不用考虑它是怎么进行划分的，sharding jdbc在执行sql的时候会对逻辑表进行解析重写。

2.1.1 真实表

指的是在数据库中真实存在的表。上面例子中，my_table0和my_table1就是真实表。

2.1.2 数据节点

由数据源.真实表组成。比如ds0.my_table0。

2.1.3 ==绑定表==

在关联查询或者子查询中，主表和子表的分片规则是一样的，那么这两张表互为绑定表。

绑定表在查询的时候， 所有路由计算将会只使用主表的策略，即根据主表的分片规则计算出要去哪些实体表查询。

由于绑定表的分片规则是一样的，所以它们的数据会按照相同的分片键被分到相同的真实表里面，所以在查询的时候能够避免去到多个表查询记录，也就是避免笛卡尔积关联。

2.2 分片相关

2.2.1 分片键

分片键表示分片的时候按照那些列来进行分片，比如按照uid列来进行分片。

当指定了分片键之后，如果是查询语句，sharding jdbc会从查询条件提取分片键，然后交由相应的分片策略进行处理，路由到相关的库和表中。

分片键包括：

  1. 分库的分片键。调用相应的分片策略，对分片键进行计算处理，最后决定分到哪一个库中。
  2. 分表的分片键。调用相应的分片策略，对分片键进行计算处理，最后决定分到哪一个表中。

分片键可以指定一个或多个。在sharding jdbc中，可以把分片键的作用理解成，需要从依据sql语句中抽取那些字段进行路由。

2.2.1 分片算法

分片算法可以对分片键进行计算处理，然后进行路由。

sharding jdbc没有具体的分片算法实现，只是提供了一些接口。根据不同的应用场景，开发者可以挑选合适的接口进行实现。

接口如下：

  1. PreciseShardingAlgorithm， 精确分片算法。
    • 处理一个分片键， 查询条件是=或者IN的情况
  2. RangeShardingAlgorithm， 范围分片算法。
    • 处理一个分片键， 查询条件是BETWEEN AND的情况
  3. ComplexKeysShardingAlgorithm， 复合分片算法。
    • 处理多个分片键，查询条件可以是=或者in
  4. ==HintShardingAlgorithm，Hint分片算法==

2.2.2 分片策略

分局不同的应用场景，需要指定不同的分片策略。比如需要对一个分片键，当查询条件是=或者in或者between的操作进行路由时，可以指定StandardShardingStrategy策略。

分片策略可以理解成是分片算法+策略。在指定一个分片策略的时候，需要指定相应的分片键和分片算法。在路由的时候会调用指定的算法进行处理。

分片策略列举如下：

  1. StandardShardingStrategy， 标准分片策略。一般用的也是这个策略。
    • 适用于一个分片键， 需要处理=, IN和BETWEEN AND的操作。
  2. ComplexShardingStrategy， 复合分片策略。
    • 适用于一个或者多个分片键，需要处理 =, IN的操作。
    • 在4.0.0-RC1版本之前，复杂的分片算法不支持between and的操作。
  3. InlineShardingStrategy， 行表达式分片策略。
    • 适用于一个分片键，分片算法简单的情况，并且操作只能是 =, IN。
  4. ==HintShardingStrategy， Hint分片策略。==
    • 通过Hint而非SQL解析的方式分片的策略

对于同一个逻辑表，只能使用一个分片策略。

3. sharding jdbc原理分析

sharding jdbc 是对 JDBC 相关的类进行了实现，比如提供了 ShardingDataSource， *ShardingConnection， *ShardingPreparedStatement 等类，JDBC在运行的时候，实际调用的是Sharding JDBC的实现。

比如 ShardingPreparedStatement的execute方法就已经包含了sql解析，分片路由相关的工作

Sharding JDBC的工作流程包括：

  1. sql解析
  2. 分片路由
  3. sql改写
  4. sql执行
  5. 结果合并

3.1 sql解析

sql解析是指，将一条sql语句的各个部分进行拆分，比如识别select或者insert关键词，查询条件是=还是in等等。

3.1.1 sql解析的缓存机制

在解析sql语句的时候，如果有设置缓存参数，那么sharding jdbc回对结果进行缓存。从而下次对相同的sql语句进行解析的时候，会直接取缓存中的值。

3.2 分片路由

分片路由主要是根据sql解析的结果，调用相应的分片算法，对分片键进行处理，得到需要路由过去的数据源集合和真实表的集合。

比如根据分片键my_id处理之后，得到的数据源集合是ds0, ds1， 真实表是 my_table0, my_table1。也就是说需要到ds0和ds1中去查询my_table0, my_table1。

如果查询的时候没有使用分片字段的查询条件，那么路由的时候将不会提取到分片键，无法进行计算。那么将会路由到所有的数据源和真实表。

3.4 结果归并

sharding jdbc需要将从不同的数据库，不同的表查询得来的结果合并成一个结果。根据不同的操作，合并操作的复杂度有所不同。

比如，如果只是一个简单的查询，那么只需要将结果直接合并成一个结果集即可。但是如果是有排序的操作，每个结果集都是各自数据源中的数据排序的，如果需要合并，还需要对这些结果集再进行排序。

结果归并根据操作的不同可以做如下的划分：

  • 遍历归并
  • 排序归并
  • 分组归并
  • 聚合归并
  • 分页归并

即排序，分组，分页，聚合函数的操作，都会涉及到复杂的归并操作。

4. 读写分离

读写分离指的是，将写入的操做和读取的操作在不同的数据库中进行。这样可以避免读操作受写操作的影响，提高效率。

对于读写分离来说，主库和从库的表结构和表数据都是一致的。
