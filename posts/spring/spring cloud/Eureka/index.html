1 .Eureka

Eureka是一个服务的注册中心，提供服务的注册和发现，以及服务的故障转移。
比如各个微服务可以注册地址信息等到Eureka，然后调用方直接到Eureka获取到这些地址信息，就可以直接访问了,当服务的地址改变了，调用方依旧可以从eureka获取的到。

Eureka分为服务端和客户端：

  1. 服务端用于维护注册的信息
  2. 客户端用于注册信息到服务端，提供方便用户调用的api，定期向服务端发送心跳（默认30秒），如果服务端没有收到心跳，将会移除这个服务节点*（默认九十秒之后移除）

Eureka和zookeeper比较

  1. Eureka是遵守AP原则，zookeeper是遵守CP原则
    • 比如Eureka各节点平等，都可以提供服务。
    • zookeeper是主从模型，如果主节点挂掉了，还需要重新选择才可提供服务。另外，当数据改变的时候，zookeeper使用的是zab协议，只有过半的节点投票成功，外部才能访问的到数据。
  2. Eureka主要用于服务的注册与发现。而zookeeper可以完成更多的功能，它是一个分布式的协调服务，提供的功能包括分布式锁，配置管理，主从选举，服务注册和发现等等。

2. spring cloud 结合Eureka server

spring cloud 引入了Eureka作为注册中心，并且提供了一些starter，方便使用Eureka。

2.1 引入依赖。

因为spring cloud时基于spring boot之上的，所以需要引入

  1. spring boot的依赖。
  2. spring cloud starter
  3. eureka server的依赖

2.2 编写配置文件

  1. 如果是启动eureka server注册中心，则需要将register-with-eureka和fetch-registry都配置为false。如果是注册服务则不需要配置这两个属性。

2.3 编写启动类

主要是加了 @EnableEurekaServer 注解用于启动Eureka server

  1. 引入eureka client的依赖， 该依赖已经包含了spring cloud starter
  2. 引入spring web starter， 因为如果需要暴露restful http风格的服务，则需要该依赖
  3. 引入spring actuator 工具包， 服务启动之后，会自动暴露一个/actuator/info的 api，用于查询跟项目相关的额外描述信息。这些信息需要actuator工具读取，详细可以参考spring boot

spring cloud有两种调用服务的方式， restTemplate和feign。所以先了解这两个工具。

4.2 Ribbon

Ribbon doc

Ribbon是Netflix提供的客户端负载均衡模块。即在调用方调用服务的时候，可以使用Ribbon来设置负载均衡的策略。

Ribbon有三个子项目。其中一个项目是为Eureka客户端实现了负载均衡的api，所以Eureka客户端在调用服务的时候可以利用Ribbon来进行负载均衡。

在Spring cloud中，默认继承的注册中心就是Eureka，所以Spring也是默认使用了Ribbon。

4.2.2 Nginx 和 Ribbon的区别

Ribbon属于客户端 (注意这里的客户端不是指浏览器，移动端等等，它也是属于后端的服务，比如Eureka Client， zuul网关等等) 的负载均衡工具。是在客户端访问服务的时候，在客户端自己本地决定访问哪台服务器。它的应用场景通常是需要在客户端进行负载均衡的，比如Dubbo中的RPC。

Nginx属于服务端的负载均衡服务器。客户端先请求到nginx服务器，nginx再通过负载均衡的策略，将请求路由到相应的后台服务器。它的应用场景通常是分发终端的访问请求，比如浏览器的请求，移动端的请求，这些请求很少会由自己去实现负载均衡的功能， 所以交由nginx服务器，再分发到各个后台服务。

Feign项目地址

5. Hystrix

==hystrix工作原理==

hystrix 文档

Hystrix用于提供超时机制，限流，熔断，降级的工具。

实际上Hystrix官方已经停止更新了，可以替代的方案时resilience4j或者Alibaba Sentinel

5.1 Hystrix 流程

  1. 在调用一个服务的时候，调用方利用hystrix会先去判断熔断器是否开启，开启了就说明服务不可用被熔断了，直接调用fallback进行降级。
  2. 如果熔断器没有开启，说明服务可用，会进一步利用限流器的判断是否可以使用线程，如果不行，就直接fallback降级。如果可以，调用远程服务。
  3. 如果远程服务执行失败，就进行fallback降级。调用成功则返回成功结果。

5.2 进一步分析熔断的过程

  1. 当在一定时间内失败的次数达到一定的阈值，熔断器就会打开
  2. 熔断器打开之后，所有的请求都不会经过调用的方法，直接返回
  3. 熔断器打开一段时间之后，就会处于半开的状态。此时允许有一个请求调用，如果调用成功，则熔断器关闭。如果失败，则重新进入打开状态。

5.3 hystrix 超时，限流，熔断，降级

  1. hystrix限流 （线程隔离） ：指的是，对于每个接口，分配各自的线程池，每一次调用该接口，就相等于占用了一个线程。当线程沾满了，后续再有调用将不成功，相当于限制了线程的访问量。
    1. 限流能够针对各个接口进行访问量的限制，避免过度的占用线程资源，影响到其它接口的调用
  2. 熔断。指的是调用接口失败次数达到一定的阈值之后，该接口将不可访问。失败指的是超时或者远程服务执行失败。

5.3.1 它们之间的关系

降级是客户端因为熔断，超时，限流，或者执行远程服务失败时，所做的一种补偿操作。

超时和远程服务执行失败次数达到一定的时候就会触发熔断，可以说超时和远程服务执行失败是导致熔断的原因。

7. 自我保护机制

7.1 心跳

Eureka client默认会每隔30秒 (eureka.instance.lease-renewal-interval-in-seconds) 向Eureka server发送心跳，并且默认如果90秒 (eureka.instance.lease-expiration-duration-in-seconds) 之内没有向server发送心跳，Eureka Server可以将其标记成可以剔除的目标。

7.2 自我保护机制原理

当Eureka server 在默认心跳时间没有收到 Eureka client端的心跳时， 并不会立即删除Eureka client的注册信息，依旧还保留着。
这是考虑到网络暂时出现网络分区的情况，但其实Eureka Client还是运行着。

当Eureka server 在过期时间内依旧没有收到心跳 （默认是90秒，由客户端设置）*，可以将其标记为可剔除，在下一次剔除的线程运行的时候 *（默认60秒运行一次） 就会真正的将Eureka client删除了。

需要注意的是，因为自我保护机制的存在，Eureka server会为微服务保留一段时间的注册信息，而且客户端访问的时候依旧可以被路由到该微服务，导致访问失败的情况。所以客户端在访问的时候，可以采取重试，超时，降级的方式处理访问失败的场景。

7.3 自我保护机制设置

自我保护机制的设置需要在客户端和服务端都进行设置：

  1. 客户端。设置心跳周期以及心跳过期时间
  2. 服务端。关闭自我保护机制，设置清理周期等。

自我保护机制默认是开启的状态，它由下面的参数指定。开启的状态下，使用默认配置，比如清除服务的线程每隔60运行一次。
